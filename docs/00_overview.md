# Project Overview: QLoRA Fine-Tuning of Llama 3.2 3B on Alpaca

## What this project does

This project takes Meta's pre-trained `Llama-3.2-3B-Instruct` language model and teaches it to better follow instructions by training it on 52,000 question-answer examples. The technique used — QLoRA — is specifically designed to do this on a single consumer GPU (like an RTX 4090) that would otherwise run out of memory trying to train a model this size. The result is a small set of "adapter" weights that can be uploaded to HuggingFace Hub so anyone can use the improved model.

---

## The Full Pipeline

```
1. prepare_data.py   — Sanity check: does the data look right? Any truncation risk?
2. evaluate.py --stage before   — Measure the base model (the starting line)
3. train.py          — Fine-tune with QLoRA, save adapter weights
4. evaluate.py --stage after    — Measure the fine-tuned model (the finish line)
5. push_to_hub.py    — Publish adapter + auto-generated README to HuggingFace
```

Each step is independent and runnable by itself, but they are designed to be run in this exact order. The evaluation numbers from steps 2 and 4 are saved as JSON files and then injected directly into the README that gets published in step 5.

## Key Concepts Explained

### What is a language model?

A language model is a function that takes a sequence of text tokens (words or word-pieces) and predicts what comes next. Llama 3.2 3B has 3 billion numerical parameters that were tuned during pre-training to make good predictions across a huge amount of internet text. After pre-training, the model has broad knowledge but may not reliably follow human instructions.

### What is fine-tuning?

Fine-tuning is the process of continuing to train a pre-trained model on a smaller, more specific dataset. Think of it like hiring an expert engineer and then putting them through a one-week onboarding program for your specific company: they already know the fundamentals, you just shape their behavior for your context. Here, we fine-tune on Alpaca — 52,000 instruction-following examples — to make the model a more reliable instruction follower.

### What is LoRA?

LoRA (Low-Rank Adaptation) is a way to fine-tune a huge model without updating all of its parameters. Imagine the model's learned knowledge is stored in enormous matrices of numbers. LoRA says: instead of changing those matrices directly (expensive), add a small pair of "side-car" matrices next to each one. During training, only these small side-cars are updated; the original matrices stay frozen.

The math: a weight matrix `W` (shape `d × d`) gets two tiny companions `A` (shape `d × r`) and `B` (shape `r × d`), where `r` (the "rank") is much smaller than `d`. The total new weight is `W + B·A`. With `r=16`, the side-car for a 4096×4096 matrix requires only `2 × 4096 × 16 = 131,072` numbers instead of `16,777,216`. That is a ~128× reduction in trainable parameters.

A useful analogy: LoRA is like adding Post-it notes to a textbook instead of printing a whole new edition. The original text is untouched; the notes add or clarify meaning.

### What is QLoRA?

QLoRA (Quantized LoRA) extends LoRA by also compressing the frozen base model weights using 4-bit quantization (specifically NF4 — Normal Float 4). Quantization is like rounding every number in the model to the nearest value on a much shorter ruler. Normally a number is stored as a 16-bit float (65,536 possible values). NF4 stores it in 4 bits (16 possible values). This reduces memory by approximately 4×.

The tradeoff: doing math in 4-bit is imprecise, so computations during training are "dequantized" back to bfloat16 on the fly. You pay a small speed cost to get a large memory savings.

Combined effect: a 3B model that normally requires ~6GB of GPU VRAM can be fine-tuned in ~4–5GB with QLoRA.

### What is the Alpaca dataset?

The Alpaca dataset (`tatsu-lab/alpaca`) contains 52,000 examples, each with three fields:
- `instruction`: a task description ("Summarize the following article…")
- `input`: optional context text (the article body), sometimes empty
- `output`: the expected response

These examples were generated by feeding diverse instructions into GPT-3.5 (`text-davinci-003`). They cover a wide range of task types: creative writing, classification, question answering, rewriting, and more.

### What is perplexity?

Perplexity measures how "surprised" the model is by a text it hasn't seen before. Technically it is `exp(average negative log-likelihood per token)`. A perplexity of 5 means: on average, the model behaves as if it had to choose among 5 equally likely next tokens. Lower is better — a perplexity of 2 means the model is very confident; a perplexity of 100 means it is quite confused.

After fine-tuning on Alpaca-style instruction-following text, we expect perplexity on the held-out Alpaca test split to decrease, because the model has learned to assign higher probability to these kinds of responses.

Perplexity is fast to compute: it only requires one forward pass (no generation), so we can run it on the full test split.

### What is ROUGE-L?

ROUGE-L (Recall-Oriented Understudy for Gisting Evaluation, Longest common subsequence variant) measures how much a generated text overlaps with a reference answer. It finds the longest sequence of words that appear in both the generated and reference texts (in order, but not necessarily contiguous) and computes an F1 score.

Example: if the reference is "the quick brown fox" and the model generates "the brown fox jumped", the LCS is "the brown fox" (3 of 4 words), giving a ROUGE-L of 0.75 (roughly).

ROUGE-L ranges from 0 (no overlap) to 1 (perfect match). It is slow to compute because it requires actually generating text for each example — hence it is only computed on 200 sampled rows.

After fine-tuning, we expect ROUGE-L to improve, because the model generates responses that more closely resemble the reference Alpaca outputs.

## How All the Pieces Fit Together

```
configs/
  train_config.yaml      ← Single source of truth for all settings

src/
  data_utils.py          ← Dataset loading, splitting, prompt formatting
  model_utils.py         ← Model/tokenizer loading, LoRA/BnB configs
  eval_utils.py          ← Perplexity and ROUGE-L metric computation

scripts/
  prepare_data.py        ← Imports src/data_utils → validates data before training
  train.py               ← Imports src/data_utils + src/model_utils → runs SFTTrainer
  evaluate.py            ← Imports all three src/ modules → measures before and after
  push_to_hub.py         ← Imports src/model_utils → publishes to HuggingFace Hub

results/
  before_finetune.json   ← Written by: evaluate.py --stage before
  after_finetune.json    ← Written by: evaluate.py --stage after

outputs/
  checkpoints/           ← Written by: train.py (mid-training checkpoints)
  final_adapter/         ← Written by: train.py (final LoRA adapter weights)
```

The `src/` modules are pure libraries — they contain no `main()` functions and are only imported, never run directly. The `scripts/` files are runnable entry points that orchestrate the `src/` functions. This separation keeps the logic testable and reusable.

The config file is loaded fresh at the start of every script. This means every script always reads the same settings — changing one YAML value affects all downstream scripts automatically.

## Data Flow Through the System

```
HuggingFace Hub (tatsu-lab/alpaca)
        │
        ▼
  load_and_split()                ← seeds reproducible 95/5 train/test split
        │
        ├──► train split (49,400 rows)
        │         │
        │         ▼
        │   format_alpaca_row()   ← wraps each row in chat template + assistant answer
        │         │
        │         ▼
        │   SFTTrainer.train()    ← updates only LoRA adapter weights
        │         │
        │         ▼
        │   outputs/final_adapter/
        │
        └──► test split (2,600 rows)
                  │
                  ├──► compute_perplexity()   ← full split, forward pass only
                  │
                  └──► compute_rouge_l()      ← 200 sampled rows, greedy generation
                              │
                              ▼
                    results/before_finetune.json
                    results/after_finetune.json
                              │
                              ▼
                    push_to_hub.py → HuggingFace Hub
```

The test split is never seen during training — it is only used by `evaluate.py`. Both train and evaluate use the same `load_and_split()` call with the same seed, guaranteeing they always get the identical rows in each split regardless of which machine or time they run on.
